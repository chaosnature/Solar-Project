1. System Overview

Two-string home solar battery system. Each string has 2x JK-BMS units managed by one YamBMS ESP32-S3 (AtomS3-Lite) over BLE. YamBMS aggregates BMS data and sends PYLON V2 CAN bus frames to a Solis 3kW inverter. Home Assistant (on Pi) monitors and controls everything.

3. Coulomb Drift Fix
ℹ️
Both strings are running the drift fix. String-1 has top + bottom anchors. String-2 has top + bottom + experimental mid anchor. The fix is inlined directly in each YAML — not as an external package.

Problem

JK-BMS relies solely on coulomb counting and only recalibrates SoC from voltage when the total_battery_capacity setting is changed. After many cycles the counter drifts — you get readings like 85% when the battery is genuinely full. JK have been asked to fix this for years and refused, instead suggesting purchasing newer hardware.

Fix Mechanism

When anchor conditions are met, the ESPHome lambda toggles the BMS capacity number: 95Ah → 94Ah (triggers JK recalibration from voltage), then 94Ah → 95Ah (restores correct value) 10 seconds later via a second interval. No delay() used — prevents watchdog panics.

JK-BMS recalibrates SoC from voltage when capacity is toggled — it does NOT reset to 100%. This means anchoring at top and bottom (steep parts of LFP curve) is safe and accurate. Mid-cycle anchoring is unreliable due to the flat LFP curve between 20–80% SoC.

Anchor Points

AnchorTrigger ConditionsReset ConditionStrings
TOP V ≥ 56.4V AND I ≤ 2A AND SoC ≥ 98% held for 30s (6 × 5s checks) V < 55.0V String-1 + String-2
BOTTOM V ≤ 48.0V AND I ≥ -2A AND SoC ≤ 5% V > 50.0V String-1 + String-2
MID (experimental) Avg cell V 3.68–3.72V AND |I| ≤ 1A (resting) — fires once per rest event Avg cell V < 3.60V or > 3.80V String-2 only

Globals Structure

# Per BMS (bms1_ and bms2_ in each YAML)
globals:
  - id: bms1_soc_anchor_counter   # counts 5s intervals at top
    type: int
  - id: bms1_soc_anchored         # top anchor fired flag
    type: bool
  - id: bms1_restore_pending      # triggers 10s restore interval
    type: bool
  - id: bms1_mid_anchored         # mid anchor fired flag (String-2 only)
    type: bool

5. How to Flash a Device
🚨
FIRST RULE every session: Pull firmware from both chips BEFORE doing anything else. Verify chip MAC and IP before flashing. Never flash blindly.

OTA Flash (standard — chip on network)

# String-1
esphome upload YamBMS_STRING1_ORIGINAL.yaml --file "0_BACKUP\firmware\yambms-1-Jun2026-v1.5.8-KITCHEN-WORKING.bin" --device 192.168.0.101

# String-2
esphome upload YamBMS_STRING2_ORIGINAL.yaml --file "0_BACKUP\firmware\yambms-2-Jun2026-v1.5.8-SHED-WORKING.bin" --device 192.168.0.21

USB Factory Flash (recovery — chip on desk via COM7)

python -m esptool --port COM7 write-flash 0x0 0_BACKUP\firmware\yambms-1-Jun2026-v1.5.8-KITCHEN-WORKING.factory.bin
🚨
The Oct 2025 OTA bin (yambms-1-Oct2025-v1.5.8-PI-LIVE.bin) crashes String-1 when flashed OTA. Use factory flash only for that file. The Oct 2025 String-2 equivalent is fine.

After Compile — Back Up Before Flashing

# Always back up the new bin before flashing
Copy-Item ".esphome\build\yambms-1\.pioenvs\yambms-1\firmware.ota.bin" "0_BACKUP\firmware\yambms-1-DATESTAMP-DESCRIPTION.bin"
Copy-Item ".esphome\build\yambms-1\.pioenvs\yambms-1\firmware.factory.bin" "0_BACKUP\firmware\yambms-1-DATESTAMP-DESCRIPTION.factory.bin"

Check Logs After Flash

6. Compile Pipeline
⚠️
Never run esphome compile directly. Always use compile.bat which runs patch_and_compile.ps1. Without the patches the compile either fails or produces a binary from wrong package versions.

Working Directory

C:\Users\osagk\Documents\0_My_Projects\Solar_Project\AI_Assist_Project\

Compile Command

# String-1
compile.bat YamBMS_STRING1_ORIGINAL.yaml

# String-2
compile.bat YamBMS_STRING2_ORIGINAL.yaml

What patch_and_compile.ps1 Does

  1. Pins all remote package refs to ref: 1.5.8 (prevents drift to main/v1.6.0)
  2. Sets refresh: 365d on all packages (prevents ESPHome re-downloading cached packages on every compile)
  3. Patches all cached bms_sensors_JK_BLE_standard.yaml files: pins esphome-jk-bms@2.5.0 (fixes errors_bitmask removal) and injects id: bms${bms_id}_total_capacity_number
  4. Runs git add . && git commit to fix CMake head-ref error during PlatformIO build
  5. Creates head-ref files for all build directories
  6. Runs esphome compile
🚨
A Gemini-generated script once ran esphome config first, which re-downloaded packages before patching — overwriting pinned versions and producing a non-functional binary. Never run esphome config before patching.

esphome-yambmsref: 1.5.8v1.6.0 has unresolved charge/discharge issue
esphome-jk-bms (syssi)@2.5.0v3.0.0 removed errors_bitmask sensor — breaks compile
refresh365dPrevents ESPHome re-downloading on every compile

Why Local Drift Fix Cannot Be a Remote Package

ESPHome fetches remote packages from GitHub. Local files cannot be included via the files: syntax in a remote package block — it will error with "does not exist in repository". The drift fix is therefore inlined directly as globals: and interval: blocks in each YAML.

Manual Backup via Command Line

The HA UI backup may fail after major version updates or when the database is locked. Use the supervisor CLI instead.

Prerequisites — Check HA is Fully Up

# Confirm both containers are running before attempting backup
sudo docker ps | grep -E "raspberrypi4.*homeassistant|hassio_supervisor"

# Expected output:
# raspberrypi4-64-homeassistant:2026.x.x   Up X minutes   homeassistant
# aarch64-hassio-supervisor:latest          Up X minutes   hassio_supervisor

Trigger Backup

# Create a full backup via supervisor CLI
sudo ha backups new --name "Manual_backup_2026-06-28"

# Success response looks like:
# Processing... Done.
# job_id: 8b563db56481416496967b12f3592593
# slug: f50fba6d

Verify Backup Created

# Check backup file appeared
ls -lh /usr/share/hassio/backup/

# Watch it grow in real time
watch -n 10 'ls -lh /usr/share/hassio/backup/'
⚠️
If backup fails with "Could not lock database": the recorder repack is still running and has the db locked. Wait 10–15 minutes and retry. If still stuck after hours, reboot the Pi with sudo reboot, wait 4 minutes for HA to fully start, then retry the backup command.
🚨
Do NOT use these API endpoints — they fail on HA 2026.x:
POST /api/services/backup/create → 400 Bad Request
POST /api/services/hassio/backup_full → 500 Internal Server Error
POST /api/backup/generate → 404 Not Found
Use sudo ha backups new instead.

9. v1.6.0 Known Issue — UNRESOLVED
🚨
Do NOT suggest upgrading to v1.6.0 as a fix for anything. Root cause not yet identified. Stay on v1.5.8.

Symptoms

  • Charge Status: Cut-Off immediately on boot
  • CCL Derating Reason: Charge Disabled
  • DCL Derating Reason: Discharge Disabled
  • Requested Charge Current: 0A
  • Requested Discharge Current: 0A
  • Battery neither charges nor discharges

What Was Tried

  • Added restore_mode: RESTORE_DEFAULT_ON via !extend — did not resolve
  • Manually turning on automatic charge/discharge switches via HA — did not persist

Suspected (uninvestigated)

  • Float/EOC charge state machine in v1.6.0 may be blocking discharge
  • NVS stores switch OFF state across OTA flashes

Current Decision

Spare chip (MAC 34:B7:DA:54:FD:18) is stored on v1.6.0 for future investigation. Both production strings remain on v1.5.8 indefinitely until root cause is found.

10. Known Issues
🔴
Undervoltage incident (while on holiday): String batteries dropped to 2.7V/cell — below Cell UVP (3.1V). YamBMS lost connection with Solis during the event; Solis may have continued discharging without BMS oversight. Root cause under investigation. Cells may have long-term capacity damage.
⚠️
BMS4 Cell 13 low voltage: historically showed ~59mV delta vs other cells. Monitor for recurrence — may indicate a weak cell or poor connection.
⚠️
Display FAILED (String-1 chip): AtomS3 display shows "display is marked FAILED: unspecified" — GPIO33/34 PSRAM conflict. Non-critical, CAN bus and BLE working fine.
ℹ️
Mid anchor reliability: The 3.7V mid anchor on String-2 is experimental. LFP voltage curve is flat between 20–80% SoC — a 0.1V measurement error could map to 40–60% SoC. Monitor carefully before applying to String-1.
